# כלי נתונים לחשבשבת

## הוראות

> **חשוב, השתמשו בייצוא הרשמי OPENFORMAT/BKMV ולא בפענוח בינארי ישיר.**
>
> חשבשבת לא מפרסמת מיקומי בתים ציבוריים לקבצים הפנימיים שלה (`.dat` / `.hsh` / `.mdb`). הייצוא המתועד והמחויב על ידי רשות המסים מכל תוכנת הנהלת חשבונות בישראל הוא **OPENFORMAT (קובץ אחיד / BKMV)**. הייצוא מפיק קובץ ZIP של שלושה קבצים: `INI.TXT` (תמצית ההפקה, מכיל את רשומת `A000` המובילה ורשומות סיכום עם ספירות בקרה לכל סוג רשומה), `BKMVDATA.TXT` (נתוני העסק, עם רשומת פתיחה אחת `A100`, אחר כך רשומות `C100`, `D110`, `D120`, `B100`, `B110`, `M100`, ואז רשומת סגירה אחת `Z900`), וקובץ `README.TXT` (פרטי הפקה כלליים למשתמש). רק `INI.TXT` ו-`BKMVDATA.TXT` נשלחים לרשות המסים או לרואה החשבון. מפרט: <https://www.misim.gov.il/TmbakmmsmlNew/Files/horaot_131.pdf>. מדריך ייצוא BKMV של חשבשבת: <https://downloads.h-erp.co.il/files/general/bkmv7-erp.pdf>. אמתו את הפלט מול סימולטור רשות המסים: <https://www.misim.gov.il/TmbakmmsmlNew/frmCheckFiles.aspx>.
>
> מפות העמודות ברוחב קבוע למטה (HESHIN_COLUMNS / PKUDOT_COLUMNS) הן **הערכות ניחוש להתקנות Windows ישנות**, לא מפרט מוסמך. השתמשו בהן רק כגיבוי כשאין ייצוא OPENFORMAT זמין, ולעולם אל תציגו אותן כפורמט החשבשבת הקנוני. לכל הגשה לרשות המסים, העברה לרואה חשבון או הפקת PCN874 / טופס 6111, תייצאו דרך OPENFORMAT במקום.

> **חשבשבת בענן (H-WEB / Wizcloud) ו-API ציבורי מסוג REST.** גרסת הענן של חשבשבת חושפת API ציבורי מסוג REST למסמכים, חשבונות ותנועות: <https://home.wizcloud.co.il/help/apidocument/>. השתמשו בו לסנכרון דו-כיווני שוטף מול Green Invoice / רווחית / iCount במקום העתקות קובץ חד-פעמיות היכן שאפשר.

> **הקשר מספר הקצאה SHAAM לשנת 2026.** פקודות יומן של חשבוניות מכירה שנוצרות או מיובאות דרך חשבשבת לעסקאות B2B בסכום שמעל הסף הנוכחי חייבות לשאת מספר הקצאה. הסף (ללא מע"מ): NIS 10,000 מאז ינואר 2026 ו-NIS 5,000 מאז 1 ביוני 2026. הדרישה תלוית-תאריך (הוכנסה בהדרגה עם סף יורד), ולכן באימות חשבוניות היסטוריות או מיובאות, בדקו כל חשבונית מול הסף שהיה בתוקף בתאריך שלה, ודלגו על חשבוניות מלפני תחילת הדרישה. אל תפסלו באופן גורף חשבוניות ישנות שמעולם לא נדרש להן מספר הקצאה. חשבונית מס מעל הסף ללא מספר הקצאה תקף לא ניתנת לקיזוז מע"מ תשומות אצל הצד השני, אז התייחסו למספר הקצאה חסר כשגיאת אימות קשיחה ולא כאזהרת "חסר" רכה. בצד הקנייה, קלטו ושמרו את מספר ההקצאה המודפס על כל חשבונית ספק מעל הסף והעבירו אותו לכרטסת ול-PCN874, זה מה שמגן על קיזוז מע"מ התשומות של הלקוח שלכם. לחשבשבת בענן יש אינטגרציית SHAAM מובנית בזמן אמת; גרסת ה-Windows עשויה לדרוש תהליך נפרד.

### שלב 1: זיהוי גרסת חשבשבת ופורמט הקבצים

תזהו באיזו גרסה של חשבשבת המשתמש עובד ואת פורמטי הקבצים הרלוונטיים:

- **חשבשבת ישנה (Windows, מקומית)**: מאחסנת נתונים במנוע ISAM או SQL קנייני. פענוח בינארי ישיר לא נתמך ושביר; השתמשו בייצוא OPENFORMAT המובנה במקום.
- **חשבשבת H-ERP (גרסת ה-ERP ל-Windows הנוכחית)**: קו המוצר הסטנדרטי נכון ל-2026. מייצאת OPENFORMAT/BKMV לתהליכי רשות המסים ורואה החשבון; קבצי `.dat` / `.hsh` ישנים עדיין עשויים להופיע בארכיונים.
- **חשבשבת בענן (H-WEB / Wizcloud, SaaS)**: ענן-נייטיב, מייצאת OPENFORMAT ישירות, בנוסף ל-API ציבורי מסוג REST לאינטגרציה חיה.
- "גולד" / "2000+" הם שמות מקו מוצרים ישן משנות ה-90 וה-2000; H-ERP / H-WEB הם השמות הנוכחיים.

קבצי נתונים נפוצים של חשבשבת:

| `PKUDOT.dat` | פקודות יומן | פקודות יומן | רוחב קבוע, Windows-1255 |

### שלב 2: טיפול בקידוד עברית

קבצי חשבשבת משתמשים בדרך כלל בקידוד Windows-1255 (עברית). תמירו ל-UTF-8 לפני עיבוד:

```python
import chardet

def detect_and_convert(file_path: str) -> str:
    """זיהוי קידוד והמרת קובץ חשבשבת ל-UTF-8."""
    with open(file_path, 'rb') as f:
        raw_data = f.read()

    detected = chardet.detect(raw_data)
    encoding = detected['encoding']

    # חשבשבת כמעט תמיד משתמשת ב-Windows-1255
    if encoding and encoding.lower() in ('windows-1255', 'iso-8859-8', 'hebrew'):
        encoding = 'windows-1255'
    elif encoding is None:
        encoding = 'windows-1255'  # ברירת מחדל בטוחה לנתוני הנהלת חשבונות בעברית

    return raw_data.decode(encoding, errors='replace')
```

מלכודות קידוד נפוצות:
- חשבשבת גולד תמיד משתמשת ב-Windows-1255
- חלק מהייצואים יכולים להשתמש ב-ISO-8859-8 (עברית חזותית) במקום עברית לוגית
- קבצים עם קידוד מעורב קורים כשנתונים הועתקו ממקורות אחרים
- BOM (סימן סדר בתים) יכול להופיע בייצואי CSV חדשים יותר

### שלב 3: פענוח קבצי נתונים ברוחב קבוע של חשבשבת

קבצי `.dat` של חשבשבת משתמשים בפריסת עמודות ברוחב קבוע. רוחב העמודות משתנה לפי סוג הקובץ:

פענוח הקבצים באמצעות מיקומי העמודות:

```python
def parse_fixed_width(content: str, columns: dict) -> list[dict]:
    """פענוח קובץ נתונים ברוחב קבוע של חשבשבת."""
    records = []
    for line in content.strip().split('\n'):
        if not line.strip():
            continue
        record = {}
        for field_name, (start, end) in columns.items():
            value = line[start:end].strip() if len(line) > start else ''
            record[field_name] = value
        records.append(record)
    return records
```

### שלב 4: ייצוא נתונים לפורמטים מודרניים

המרת נתונים מפוענחים של חשבשבת ל-JSON, CSV או Excel:

```python
import csv
import json

def export_to_csv(records: list[dict], output_path: str):
    """ייצוא רשומות מפוענחות ל-CSV בקידוד UTF-8 עם BOM לתאימות Excel."""
    if not records:
        return
    with open(output_path, 'w', encoding='utf-8-sig', newline='') as f:
        writer = csv.DictWriter(f, fieldnames=records[0].keys())
        writer.writeheader()
        writer.writerows(records)

def export_to_json(records: list[dict], output_path: str):
    """ייצוא רשומות מפוענחות ל-JSON עם תמיכה בעברית."""
    with open(output_path, 'w', encoding='utf-8') as f:
        json.dump(records, f, ensure_ascii=False, indent=2)

def export_to_excel(records: list[dict], output_path: str, sheet_name: str = 'Data'):
    """ייצוא רשומות מפוענחות ל-Excel עם עיצוב RTL תקין."""
    from openpyxl import Workbook
    from openpyxl.worksheet.properties import WorksheetProperties

    wb = Workbook()
    ws = wb.active
    ws.title = sheet_name
    ws.sheet_properties = WorksheetProperties(rightToLeft=True)

    # כתיבת כותרות
    headers = list(records[0].keys())
    for col, header in enumerate(headers, 1):
        ws.cell(row=1, column=col, value=header)

    # כתיבת נתונים
    for row_idx, record in enumerate(records, 2):
        for col_idx, header in enumerate(headers, 1):
            ws.cell(row=row_idx, column=col_idx, value=record.get(header, ''))

    wb.save(output_path)
```

### שלב 5: ייבוא נתונים לפורמט חשבשבת

כשמייבאים נתונים לחשבשבת, תיצרו קבצים ברוחב קבוע שתואמים את הפריסה הצפויה. **זהירות:** מפות העמודות כאן הן אותן הערכות ניחוש שסומנו בבלוק ההוראות, לא מפרט מוסמך, וייבוא נתונים ברוחב קבוע לא-מיושר לתוך חברה פעילה עלול לפגום בספרים. העדיפו את ממשק/תבנית הייבוא המתועדים של חשבשבת עצמה, ותמיד בדקו קובץ שנוצר מול עותק של קובץ החברה תחילה, לעולם לא מול הקובץ החי.

```python
def generate_hashavshevet_import(records: list[dict], columns: dict, output_path: str):
    """יצירת קובץ ברוחב קבוע לייבוא לחשבשבת."""
    lines = []
    for record in records:
        line = ''
        sorted_cols = sorted(columns.items(), key=lambda x: x[1][0])
        for field_name, (start, width_end) in sorted_cols:
            width = width_end - start
            value = str(record.get(field_name, ''))
            # ריפוד או חיתוך לרוחב מדויק
            if len(value) > width:
                value = value[:width]
            else:
                value = value.ljust(width)
            line += value
        lines.append(line)

    with open(output_path, 'w', encoding='windows-1255', errors='replace') as f:
        f.write('\n'.join(lines))
```

כללי אימות ייבוא:
- מספרי חשבון חייבים להופיע במתכונת החשבונות
- תאריכים חייבים להיות בפורמט DD/MM/YYYY (פורמט ישראלי)
- סכומים חייבים להשתמש בנקודה כמפריד עשרוני (לא פסיק)
- חשבון חובה וחשבון זכות לא יכולים להיות זהים
- מספרי מנות חייבים להיות רציפים בתוך שנת מס
- קודי מטבע חייבים להתאים לטבלת המטבעות הפנימית של חשבשבת. הקודים המספריים הפנימיים האלה משתנים בין התקנה להתקנה, אז תאמתו אותם מול טבלת המטבעות של ההתקנה שלכם במקום להניח מיפוי קבוע או להשתמש בקודי ISO 4217.

### שלב 6: העברת נתונים למערכות ענן

כשעוברים מחשבשבת לפתרונות הנהלת חשבונות בענן:

**לפני שמתחילים (קראו את זה קודם):**
- העברה או ייצוא מחשבשבת לא פוטרים מחובת השמירה של ניהול פנקסים. את מערכת החשבונות והרשומות חובה לשמור 7 שנים מתום שנת המס הרלוונטית (או 6 שנים מיום הגשת הדוח, לפי המאוחר). אל תשביתו ואל תמחקו את מערכת חשבשבת המקורית אחרי המעבר; השאירו אותה (או עותק ארכיון מלא) נגישה לכל התקופה הזו.
- היכן שמערכת היעד תומכת בכך, תייבאו את קובץ ה-BKMV במבנה אחיד ישירות (יבוא נתונים מקובץ במבנה אחיד). רווחית ומערכות ענן אחרות מקבלות את הקובץ האחיד כמו שהוא. זה עדיף על מיפוי ידני של CSV/Excel כי זה שומר על רציפות מספרי המסמכים ועל מבנה השדות שרשות המסים הגדירה.
- שמרו על רציפות מספור המסמכים (מספר עוקב) לאורך המעבר כך שכל סוג מסמך ישמור מספר רץ ללא קפיצות.
- תאמתו את מאזן הבוחן הפותח של המערכת החדשה מול מאזן הבוחן הסוגר של המערכת הישנה לפני העלייה לאוויר; תחקרו כל הפרש במקום לעגל אותו.
- קובץ ה-BKMV במבנה אחיד אינו מכיל רשומות שכר/משכורת, רק את נתוני הנהלת החשבונות והמסמכים. העברה מלאה של חברה חייבת לחלץ בנפרד את היסטוריית השכר (דרך מודול השכר של חשבשבת / תהליך הדיווח השנתי על שכר); הסתמכות על הקובץ האחיד בלבד משמיטה בשקט את נתוני השכר.

**מעבר ל-iCount:**
- תייצאו את מתכונת החשבונות ואז תמפו מספרי חשבון לקטגוריות iCount
- תייצאו חשבוניות פתוחות ויתרות לקוחות/ספקים
- iCount מקבלת ייבוא CSV עם כותרות עמודות ספציפיות

**מעבר לרווחית:**
- תייצאו יומן מלא לשנת המס הנוכחית
- תמפו סוגי חשבון של חשבשבת לסיווג החשבונות של רווחית

### שלב 6.5: הגשות לרשות המסים (PCN874 וטופס 6111)

שתי הגשות לרשות המסים מופקות בדרך כלל מנתוני חשבשבת. שתיהן מבוססות על מפרט שאינו תלוי בתוכנה, אז תפיקו אותן מייצוא OPENFORMAT ולא מפענוח בינארי מנחש:

- **PCN874 (דוח מפורט מע"מ)**: קובץ טקסט במבנה קבוע, לא רשימת חשבוניות שטוחה. הוא מתחיל ברשומת כותרת/פתיחה (מספר העוסק של בית העסק, תקופת הדיווח, וסיכומים/ספירות), ואחריה רשומות פירוט שממוינות לפי קוד סוג תנועה, עסקאות (מכירות/פלט) לעומת תשומות (קלט, ועוד סוגים מיוחדים כמו רשומון יבוא). עבור תנועת תשומות מספר העוסק של הספק הוא חובה, אחרת אי אפשר לקזז את מע"מ התשומות. דיווח מפורט מחויב רק מעל סף המחזור (לעוסק יחיד, מחזור שנתי מעל NIS 500,000 החל מ-1 בינואר 2026). שתי הקלות חלות על עוסק יחיד: (א) חשבוניות מס שסכומן לפני מע"מ הוא NIS 5,000 או פחות ניתן לדווח כסכום מצטבר אחד במקום פירוט שורה-שורה; (ב) העוסק יכול לפנות למשרד המע"מ האזורי שלו כדי לדחות את החובה ל-1 בינואר 2027 אם בדוחות 2025 שלו לפחות 90% מחשבוניות התשומות היו כל אחת NIS 5,000 או פחות. מנהל החשבונות מייצא אותו חודשי או דו-חודשי ומעלה אותו לרשות המסים, שמצליבה מע"מ תשומות מול מע"מ עסקאות. לחשבשבת יש ייצוא PCN874 מובנה. המפרט נמצא באתר רשות המסים (ראו קישורי עזר).
- **טופס 6111 (דוח התאמה למס)**: נספח לדוח השנתי שנושא נתוני רווח והפסד, מאזן ודוח התאמה למס, ומוגש מקוון. מנהל החשבונות או רואה החשבון מייצא את מאזן הבוחן מחשבשבת, ממפה כל חשבון לקודי השורות של 6111, ומגיש את הנספח. תאמתו את קודי השורות הנוכחיים מול מפרט 6111 של רשות המסים לאותה שנה (ראו קישורי עזר).

### שלב 7: אימות שלמות נתונים

אחרי כל פעולת ייבוא או ייצוא, תוודאו את שלמות הנתונים. **הסתייגות לגבי `validate_trial_balance` שלמטה:** בפורמט PKUDOT של חובה+זכות באותה שורה היא מסכמת את אותו `amount` לשני הסכומים, ולכן היא מאוזנת מעצם הבנייה ולא יכולה לתפוס היפוך סימן, סכום שגוי, ספרות שהוחלפו או צד נגדי חסר. התייחסו אליה כבדיקת עשן גסה בלבד. איזון אמיתי חייב לבוא מסימולטור בדיקת הקבצים של רשות המסים (עבור ייצוא OPENFORMAT) או מבדיקת תנועות חובה-מול-זכות ב-B100, לא מהפונקציה הזו, ואל תציגו את תוצאת ה"מאוזן" שלה לרואה חשבון כאסמכתא.

```python
def validate_trial_balance(records: list[dict]) -> dict:
    """אימות שחובות שווים לזכויות בפקודות יומן."""
    total_debit = 0
    total_credit = 0
    errors = []

    for i, record in enumerate(records):
        try:
            amount = float(record.get('amount', 0))
            if record.get('account_debit'):
                total_debit += amount
            if record.get('account_credit'):
                total_credit += amount
        except ValueError:
            errors.append(f"שורה {i+1}: סכום לא תקין '{record.get('amount')}'")

    balanced = abs(total_debit - total_credit) < 0.01
    return {
        'balanced': balanced,
        'total_debit': round(total_debit, 2),
        'total_credit': round(total_credit, 2),
        'difference': round(total_debit - total_credit, 2),
        'errors': errors,
    }

def validate_account_references(entries: list[dict], accounts: list[dict]) -> list[str]:
    """אימות שכל החשבונות המוזכרים קיימים במתכונת החשבונות."""
    valid_accounts = {a['account_number'] for a in accounts}
    errors = []
    for i, entry in enumerate(entries):
        debit_acc = entry.get('account_debit', '').strip()
        credit_acc = entry.get('account_credit', '').strip()
        if debit_acc and debit_acc not in valid_accounts:
            errors.append(f"שורה {i+1}: חשבון חובה '{debit_acc}' לא נמצא במתכונת החשבונות")
        if credit_acc and credit_acc not in valid_accounts:
            errors.append(f"שורה {i+1}: חשבון זכות '{credit_acc}' לא נמצא במתכונת החשבונות")
    return errors
```

## מלכודות נפוצות

- קבצי חשבשבת משתמשים בקידוד Windows-1255, לא UTF-8. סוכנים כמעט תמיד ינסו לקרוא את הקבצים כ-UTF-8, וזה יגרום ל-UnicodeDecodeError על התו העברי הראשון.
- פורמט התאריך בחשבשבת הוא DD/MM/YYYY (סטנדרט ישראלי). ייצואי בנק יכולים להשתמש ב-YYYY-MM-DD (ISO) או MM/DD/YYYY (ארה"ב). סוכנים עלולים לא לזהות שהפורמט לא מתאים.
- מיקומי עמודות ברוחב קבוע שונים בין גרסאות חשבשבת (גולד מול 2000+ מול חדשות יותר). סוכנים עלולים להחיל פריסת עמודות של גרסה אחת על נתונים מגרסה אחרת.
- קודי מטבע פנימיים בחשבשבת שונים מקודי ISO 4217, והמיפוי המדויק משתנה בין התקנה להתקנה. אז תאמתו אותו מול טבלת המטבעות של ההתקנה שלכם במקום להניח ערכים קבועים. סוכנים שמשתמשים בעיוורון בקודי ISO 4217 יפיקו ערכים שחשבשבת תדחה בייבוא.
- בייצוא ל-CSV בשביל Excel, הקבצים חייבים להיות בקידוד UTF-8 עם BOM (utf-8-sig). בלי BOM, Excel לא יציג תווים עבריים כמו שצריך ויראה ג'יבריש.
- ב-OPENFORMAT, לתנועה במט"ח כדאי שהרשומה תישא גם את הסכום בשקלים וגם את הסכום במטבע המקורי (המפרט מספק שדות מטבע לכך), אז אל תשמיטו את ערך המטבע המקורי בעת ההמרה. קבצי `BKMVDATA.TXT` גדולים מאוד עלולים להידחות על ידי הסימולטור או על ידי מערכת היעד, וצריך לפצל אותם לטווחי תאריכים קטנים יותר.

## פתרון בעיות

### שגיאה: "UnicodeDecodeError: 'utf-8' codec can't decode byte 0xe9"
סיבה: הקובץ מקודד ב-Windows-1255 (עברית) אבל נקרא כ-UTF-8. זו השגיאה הכי נפוצה בעבודה עם קבצי חשבשבת, כי התוכנה משתמשת ב-Windows-1255 כברירת מחדל.
פתרון: תציינו במפורש `encoding='windows-1255'` כשקוראים את הקובץ. אם אתם לא בטוחים לגבי הקידוד, תשתמשו בספריית `chardet` לזיהוי אוטומטי. לקבצים עם קידוד מעורב, תשתמשו ב-`errors='replace'` כדי להחליף תווים שאי אפשר לפענח.

### שגיאה: "מאזן הבוחן לא מאוזן (הפרש: X.XX)"
סיבה: הפרשי עיגול מהמרות מטבע, ייצואים חלקיים (פקודות חסרות ממנה), או נתונים פגומים בקובץ המקור. חשבשבת לפעמים מאחסנת סכומים עם ספרות עשרוניות נוספות באופן פנימי.
פתרון: קודם תבדקו אם ההפרש הוא שגיאת עיגול קטנה (פחות משקל אחד). אם כן, תיצרו פקודת התאמה. להפרשים גדולים יותר, תוודאו שהייצוא כולל את כל המנות לתקופה. תייצאו מחדש מחשבשבת עם אפשרות "ייצוא מלא" במקום ייצוא מסונן.

### שגיאה: "מספר חשבון לא נמצא במתכונת החשבונות"
סיבה: פקודות יומן מתייחסות לחשבונות שנמחקו או שמספרם השתנה בחשבשבת, או שייצוא מתכונת החשבונות הוא משנת מס אחרת מפקודות היומן.
פתרון: תייצאו את מתכונת החשבונות ואת פקודות היומן מאותו בסיס נתונים ושנת מס בחשבשבת. אם חשבונות מוספרו מחדש, תיצרו טבלת מיפוי ותעדכנו הפניות לפני הייבוא. תבדקו אם אפסים מובילים הוסרו תוך כדי ההמרה.

### שגיאה: "אי-התאמה בפורמט תאריך בזמן ייבוא"
סיבה: חשבשבת מצפה ל-DD/MM/YYYY (פורמט ישראלי) אבל נתוני המקור משתמשים ב-MM/DD/YYYY (פורמט אמריקאי) או YYYY-MM-DD (פורמט ISO). זה קורה בדרך כלל בייבוא נתוני בנק או נתונים ממערכות בינלאומיות.
פתרון: תנרמלו את כל התאריכים ל-DD/MM/YYYY לפני שאתם יוצרים את קובץ הייבוא. תבדקו תאריכים דו-משמעיים שבהם יום וחודש יכולים להתחלף (למשל, 03/04/2025 יכול להיות 3 באפריל או 4 במרץ) ותוודאו מול פורמט מערכת המקור.
