# התאמת בנק ישראלי

אוטומציה של תהליך התאמת עסקאות בנק ישראליות מול רשומות הנהלת החשבונות שלכם. הסקיל הזה משתמש בספריית הקוד הפתוח `israeli-bank-scrapers` כדי למשוך עסקאות ונותן לכם זרימת עבודה מסודרת להתאמה, זיהוי פערים והפקת דוחות.

## הוראות

### שלב 1: הגדרת סביבת העבודה

תתקינו את התלויות הנדרשות לסריקת בנקים ועיבוד נתונים.

```bash
npm init -y
npm install israeli-bank-scrapers csv-parse csv-stringify dayjs
```

אם אתם מתכננים לייבא קבצי OFX במקום סריקה, תתקינו גם:

```bash
npm install ofx-js
```

הספרייה הראשית מתוחזקת כאן: https://github.com/eshaham/israeli-bank-scrapers

למי שרוצה גם מעקב תקציבי אוטומטי לצד ההתאמה, שווה לבדוק את Caspion: https://github.com/brafdlog/caspion

### שלב 2: הגדרת פרטי גישה לבנק

תיצרו קובץ הגדרות לחיבורי הבנק. אל תכתוב פרטי גישה ישירות בסקריפטים.

תיצרו קובץ בשם `bank-config.json` במבנה הבא:

```json
{
  "accounts": [
    {
      "id": "main-business",
      "bank": "hapoalim",
      "credentials": {
        "userCode": "${BANK_USER_CODE}",
        "password": "${BANK_PASSWORD}"
      }
    }
  ]
}
```

מזהי בנקים נתמכים:
- `hapoalim` - בנק הפועלים
- `leumi` - בנק לאומי
- `discount` - בנק דיסקונט
- `mizrahi` - מזרחי טפחות
- `otsar-hahayal` - אוצר החייל
- `mercantile` - מרכנתיל דיסקונט
- `max` - מקס (לשעבר לאומי קארד)
- `visa-cal` - ויזה כאל
- `isracard` - ישראכרט

תשמרו את פרטי הגישה האמיתיים במשתני סביבה, לא בקובץ ההגדרות.

### שלב 3: שליפת עסקאות מהבנק

תיצרו סקריפט סריקה שמושך עסקאות לטווח תאריכים שאפשר להגדיר.

```javascript
const { createScraper } = require('israeli-bank-scrapers');

async function fetchTransactions(bankId, credentials, startDate) {
  const scraper = createScraper({
    companyId: bankId,
    startDate: startDate,
    combineInstallments: false,
    showBrowser: false
  });

  const result = await scraper.scrape(credentials);

  if (!result.success) {
    throw new Error(`הסריקה נכשלה: ${result.errorType} - ${result.errorMessage}`);
  }

  return result.accounts.flatMap(account =>
    account.txns.map(txn => ({
      date: txn.date,
      amount: txn.chargedAmount,
      description: txn.description,
      memo: txn.memo || '',
      reference: txn.identifier || '',
      status: txn.status,
      accountNumber: account.accountNumber
    }))
  );
}
```

### שלב 4: ייבוא רשומות הנהלת חשבונות

תטענו את החשבוניות והקבלות מ-CSV, או תתחברו למערכת הנהלת החשבונות שלכם. מנוע ההתאמה מצפה לרשומות בפורמט מנורמל.

פורמט רשומת חשבונאות צפוי:

| שדה | סוג | תיאור |
|------|------|--------|
| `date` | `YYYY-MM-DD` | תאריך עסקה |
| `amount` | `number` | סכום בש"ח (שלילי להוצאות) |
| `reference` | `string` | מספר חשבונית/קבלה |
| `vendor` | `string` | שם הספק או מקבל התשלום |
| `category` | `string` | קטגוריה חשבונאית |

כשמייבאים CSV, תטפלו בפורמטים ישראליים:
- סכומי שקלים יכולים להשתמש בפסיק כמפריד עשרוני (1,234.56 או 1.234,56)
- תאריכים יכולים להופיע כ-DD/MM/YYYY (פורמט ישראלי) במקום YYYY-MM-DD
- שמות עסקים בעברית דורשים קידוד UTF-8

### שלב 5: הגדרת כללי התאמה

תגדירו כללים להתאמת עסקאות אוטומטית. מנוע ההתאמה תומך בכמה אסטרטגיות שרצות לפי סדר עדיפות.

**התאמה מדויקת**: התאמה לפי מספר אסמכתא וסכום מדויק.

**התאמה מקורבת**: התאמה לפי טווח תאריכים (+/- 3 ימים), סטיית סכום (עד 1 ש"ח), ודמיון בשם הספק.

**התאמת דפוסים**: תגדירו regex לעסקאות חוזרות (שכירות, חשבונות שירותים, מנויים).

```javascript
const matchingRules = [
  {
    name: 'exact-reference',
    priority: 1,
    match: (bankTxn, accRecord) =>
      bankTxn.reference === accRecord.reference &&
      Math.abs(bankTxn.amount - accRecord.amount) < 0.01
  },
  {
    name: 'amount-date-fuzzy',
    priority: 2,
    match: (bankTxn, accRecord) => {
      const dateDiff = Math.abs(
        dayjs(bankTxn.date).diff(dayjs(accRecord.date), 'day')
      );
      const amountDiff = Math.abs(bankTxn.amount - accRecord.amount);
      return dateDiff <= 3 && amountDiff <= 1.0;
    }
  },
  {
    name: 'recurring-pattern',
    priority: 3,
    patterns: [
      { regex: /חשמל|electric/i, category: 'utilities' },
      { regex: /ארנונה|municipal/i, category: 'municipal-tax' },
      { regex: /ביטוח|insurance/i, category: 'insurance' }
    ]
  }
];
```

### שלב 6: הרצת מנוע ההתאמה

תריצו את תהליך ההתאמה ותסווגו את התוצאות לשלוש קטגוריות:
1. **מותאם** - עסקת בנק שמותאמת לרשומה חשבונאית
2. **לא מותאם בבנק** - עסקאות בנק בלי רשומה חשבונאית מתאימה (חשבוניות חסרות)
3. **לא מותאם בהנהלת חשבונות** - רשומות חשבונאיות בלי עסקת בנק מתאימה (הפקדות ממתינות או שגיאות)

בנוסף, תסמנו עסקאות חשודות:
- סכומים כפולים באותו תאריך
- עסקאות גדולות במיוחד (מעל סף שאפשר להגדיר)
- עסקאות בסופי שבוע או חגים (לוח שנה ישראלי)

### שלב 7: הפקת דוחות התאמה

תפיקו דוח מסודר שמציג את סטטוס ההתאמה.

הדוח צריך לכלול:
- **סעיף סיכום**: סה"כ מותאמים, לא מותאמים, ספירות וסכומים בכל צד
- **טבלת עסקאות מותאמות**: רשומת בנק מזווגת עם רשומה חשבונאית
- **עסקאות בנק לא מותאמות**: ממוינות לפי סכום בסדר יורד לתעדוף
- **רשומות חשבונאיות לא מותאמות**: רשומות שצריך לבדוק
- **פריטים חשודים**: רשומות מסומנות שדורשות בדיקה ידנית
- **השוואת יתרות**: יתרת סגירה בבנק מול יתרת ספר חשבונות, עם הפרש ההתאמה

פורמטי פלט:
- CSV לייבוא לתוכנות גיליונות אלקטרוניים
- JSON לעיבוד בקוד
- סיכום קונסולי לבדיקה מהירה

## דוגמאות

### דוגמה 1: התאמה חודשית לעסק קטן

המשתמש אומר: "התאם את חשבון העסק שלי בבנק הפועלים לינואר 2026 מול הייצוא מ-QuickBooks."

פעולות:
1. שליפת עסקאות מבנק הפועלים לתאריכים 1-31 בינואר 2026 עם `israeli-bank-scrapers`
2. פענוח קובץ ה-CSV מ-QuickBooks, נרמול תאריכים מ-DD/MM/YYYY ל-YYYY-MM-DD וסכומים לש"ח
3. הרצת כללי התאמה: התאמה מדויקת לפי אסמכתא קודם, ואז התאמה מקורבת לפי תאריך+סכום
4. זיהוי 142 עסקאות מותאמות, 8 עסקאות בנק לא מותאמות, ו-3 רשומות חשבונאיות לא מותאמות

תוצאה: דוח התאמה שמציג שיעור התאמה של 98%. 8 הרשומות הלא מותאמות בבנק הן משיכות מזומן מכספומט בלי קבלות. 3 הרשומות החשבונאיות הלא מותאמות הן צ'קים שעוד לא נפרעו. הדוח מדגיש חיוב כפול חשוד של 2,450 ש"ח אצל אותו ספק באותו תאריך.

### דוגמה 2: התאמת ריבוי בנקים עם כרטיסי אשראי

המשתמש אומר: "אני צריך להתאים גם את חשבון העו"ש שלי בלאומי וגם את כרטיס האשראי מקס מול מערכת הנהלת החשבונות שלי לרבעון 4 של 2025."

פעולות:
1. שליפת עסקאות מבנק לאומי (אוקטובר-דצמבר 2025) ומכרטיס אשראי מקס לאותה תקופה
2. טעינת רשומות חשבונאיות מייצוא CSV מסונן לרבעון 4
3. הרצת התאמה בנפרד לכל חשבון, ואז הפקת סיכום משולב
4. סימון עסקאות תשלומים (תשלומים מפוצלים) שמופיעות כרשומה אחת בהנהלת חשבונות

תוצאה: דוח התאמה משולב שמכסה את שני החשבונות. עו"ש לאומי מציג שיעור התאמה של 97% עם 12 פריטים לא מותאמים. כרטיס אשראי מקס מציג שיעור התאמה של 91%, כשרוב הפריטים הלא מותאמים הם פיצולי תשלומים. הדוח כולל המלצה לפצל 4 רשומות חשבונאיות כדי להתאים לדפוס התשלומים. הפרש התאמה כולל: 127.50 ש"ח, שמקורו בהפרש עיגול של המרת מטבע חוץ.

### דוגמה 3: זיהוי חשבוניות חסרות לפני הגשת דוח מס

המשתמש אומר: "אני צריך למצוא את כל עסקאות הבנק מ-2025 שאין להן חשבוניות תואמות לפני הגשת דוח המס השנתי."

פעולות:
1. שליפת שנה שלמה של עסקאות מבנק דיסקונט ל-2025
2. טעינת מרשם החשבוניות המלא מייצוא מערכת הנהלת החשבונות
3. התאמה של כל העסקאות, עם דגש על זיהוי חיובי בנק לא מותאמים (הוצאות בלי חשבוניות)
4. קיבוץ של עסקאות לא מותאמות לפי ספק וקטגוריה

תוצאה: נמצאו 34 עסקאות הוצאה בסך 18,200 ש"ח בלי חשבוניות תואמות. הפערים הגדולים הם: ציוד משרדי מספק שחייב במזומן (6 עסקאות, 3,400 ש"ח), מנויי תוכנה שחויבו בדולר והומרו (8 עסקאות, 5,100 ש"ח), וחיובי חניה/כבישי אגרה (20 עסקאות קטנות, 9,700 ש"ח). מתקבלת רשימה מתועדפת לרואה החשבון לאיתור או יצירת חשבוניות חסרות.

## מלכודות נפוצות

- לכל בנק ישראלי (לאומי, הפועלים, דיסקונט, מזרחי טפחות, בינלאומי) יש פורמט ייצוא תנועות שונה. סוכנים עלולים להניח מבנה CSV אחיד בין כל הבנקים.
- תאריכי תנועות בנקאיות בישראל הם בפורמט DD/MM/YYYY. סוכנים עלולים לפרסר תאריכים כ-MM/DD/YYYY ולהחליף בשקט בין יום לחודש.
- פירעון המחאות בישראל לוקח 1-3 ימי עסקים, וזה יוצר פערי תזמון בין תאריכי הבנק לתאריכים הרשומים. סוכנים עלולים לא להתחשב בתקופות ציפה (float).
- בנקים ישראליים עובדים בשבוע עסקים ראשון-חמישי. תנועות מיום שישי או שבת מעובדות ביום ראשון. סוכנים עלולים להחיל הנחות עיבוד של שני-שישי.
- סליקת כרטיסי אשראי בישראל מגיעה כחיוב מרוכז מחברות האשראי (ישראכרט, כאל, מקס), לא כתנועות בודדות. סוכנים עלולים לנסות להתאים רכישות בודדות מול דפי בנק במקום להתאים את סכום הסליקה הכולל.

## פתרון בעיות

### שגיאה: "Scrape failed: INVALID_PASSWORD"
סיבה: פרטי הגישה לבנק שגויים, פגו תוקף, או שהחשבון דורש איפוס סיסמה. חלק מהבנקים הישראליים מחייבים החלפת סיסמה תקופתית.
פתרון: תאמתו את פרטי הגישה על ידי כניסה ידנית לאתר הבנק. אם הסיסמה שונתה לאחרונה, תעדכנו את משתני הסביבה. בבנקים שדורשים OTP או אימות דו-שלבי, תוודאו שהגדרות הסורק כוללות את השדות הנוספים הדרושים.

### שגיאה: "No transactions found for the specified date range"
סיבה: טווח התאריכים יכול להיות צר מדי, יכול להיות שאין פעילות בחשבון, או שסורק הבנק דורש פורמט תאריך שונה.
פתרון: תרחיבו את טווח התאריכים ותוודאו ש-`startDate` הוא אובייקט Date תקף ב-JavaScript. תבדקו שיש עסקאות בחשבון הבנק בתקופה שציינתם, על ידי כניסה לאתר הבנק. חלק מהסורקים מחזירים עסקאות מתאריך ההתחלה ועד היום, לא עד תאריך סיום מוגדר.

### שגיאה: "CSV parsing failed: unexpected character at position N"
סיבה: תוכנות הנהלת חשבונות ישראליות פעמים רבות מייצאות CSV בקידוד Windows-1255 (עברית) או כוללות סמני BOM שמפריעים למפענחי UTF-8.
פתרון: תמירו את הקובץ ל-UTF-8 לפני הפענוח: `iconv -f WINDOWS-1255 -t UTF-8 input.csv > output.csv`. כחלופה, תציינו את הקידוד באפשרויות מפענח ה-CSV. גם תבדקו אם יש מפרידי נקודה-פסיק (נפוץ בייצוא Excel ישראלי) במקום פסיקים.

### שגיאה: "Amount mismatch: bank shows -X but accounting shows -Y"
סיבה: הפרשי עיגול בהמרת מטבע, חישובי מע"מ, או פיצול תשלומים יכולים לגרום לפערים קטנים בין סכומי הבנק לרשומות החשבונאיות.
פתרון: תגדירו את סף סטיית ההתאמה. לסכומי שקלים, סטייה של 1.00 ש"ח מטפלת ברוב מקרי העיגול. לעסקאות שכוללות המרת מטבע חוץ, תגדילו את הסטייה ל-5.00 ש"ח. אם הפערים שיטתיים, תבדקו אם מע"מ (18% בישראל) כלול בסכומי הבנק אבל מופרד ברשומות החשבונאיות.
